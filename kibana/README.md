# Kibana

This directory contains the build context for the custom Kibana image: the Dockerfile and `kibana.yml` configuration file.

## Directory Structure

```
kibana/
  Dockerfile          # custom image build
  config/
    kibana.yml        # Kibana server configuration
```

## How Kibana Connects to Elasticsearch

Kibana is configured to connect to all three Elasticsearch nodes simultaneously:

```yaml
ELASTICSEARCH_HOSTS: ["https://es-01:9200","https://es-02:9200","https://es-03:9200"]
```

Kibana load-balances requests across the three nodes. If one node goes down, Kibana continues to function using the remaining nodes. Kibana authenticates to Elasticsearch using the built-in `kibana_system` user, whose password is set by the `elastic-setup` service on first run.

The `kibana_system` user has limited privileges by design — it can only perform operations that Kibana itself needs. It cannot be used to query data or manage indices directly. For those tasks, use the `elastic` superuser or a dedicated user with appropriate roles.

## Key Configuration Decisions

### HTTPS

Kibana itself is served over HTTPS using its own certificate generated by `elastic-setup`:

```yaml
SERVER_SSL_ENABLED: true
SERVER_SSL_CERTIFICATE: config/certs/kibana/kibana.crt
SERVER_SSL_KEY: config/certs/kibana/kibana.key
```

This means both the connection from your browser to Kibana and the connection from Kibana to Elasticsearch are TLS-encrypted end-to-end.

### Encryption Keys

Three separate encryption keys are required and must be set in `.env` before starting the stack:

```
KIBANA_ENCRYPTION_KEY    # encrypts saved objects (dashboards, index patterns)
KIBANA_REPORTING_KEY     # encrypts reporting jobs
KIBANA_SECURITY_KEY      # encrypts session cookies
```

These keys must be at least 32 characters long. If they are not set or change between restarts, Kibana will invalidate all existing saved objects and active user sessions. Generate them once and keep them stable:

```bash
# Generate a random 32-character key
openssl rand -hex 32
```

### Logging

Log level is set to `warn` in production. When debugging Kibana startup issues or integration problems with Elasticsearch, temporarily increase it:

```bash
# Inside the kibana container
curl -X POST http://localhost:5601/api/kibana/settings \
  -u elastic:$ELASTIC_PASSWORD \
  -H "kbn-xsrf: true" \
  -H "Content-Type: application/json" \
  -d '{"changes": {"logging.root.level": "info"}}'
```

## First Login

After the stack is up, access Kibana at `https://localhost:<KIBANA_PORT>` (default port is defined in `.env`).

Log in with:
- **Username:** `elastic`
- **Password:** value of `ELASTIC_PASSWORD` in `.env`

On first login, Kibana may prompt you to start a guided setup tour — this can be skipped.

## Key Areas in the UI

| Area | Path | What it is for |
|---|---|---|
| Discover | Analytics > Discover | Explore and filter raw log events |
| Dashboards | Analytics > Dashboards | Pre-built and custom visualizations |
| Dev Tools | Management > Dev Tools | Run Elasticsearch API queries directly from the browser |
| Fleet | Management > Fleet | Manage Elastic Agent policies and enrolled agents |
| Index Management | Management > Stack Management > Index Management | View indices, data streams, and component templates |
| ILM Policies | Management > Stack Management > Index Lifecycle Policies | View and edit ILM policies |
| Ingest Pipelines | Management > Stack Management > Ingest Pipelines | View, test, and edit ingest pipelines |

**Dev Tools** is particularly useful — it is the easiest way to run Elasticsearch API calls without dealing with curl, certificates, or authentication headers. Equivalent to running curl against the Elasticsearch API but from the browser.

## Health Check

The health check used by Docker waits for Kibana's status API to report `available`:

```bash
curl -sf --cacert config/certs/ca/ca.crt \
  https://localhost:5601/api/status
```

Kibana takes longer to start than Elasticsearch because it waits for all three nodes to be healthy before completing initialization. The `start_period` in the compose file is set to 150 seconds to account for this. If Kibana appears stuck, check that all three Elasticsearch nodes are healthy first.

## Useful Commands

Check Kibana status from outside the container:

```bash
curl -sf --cacert config/certs/ca/ca.crt \
  https://localhost:$KIBANA_PORT/api/status | jq '.status.overall'
```

Export all saved objects (dashboards, index patterns, visualizations) as a backup:

```bash
curl -sf --cacert config/certs/ca/ca.crt \
  -u elastic:$ELASTIC_PASSWORD \
  -H "kbn-xsrf: true" \
  "https://localhost:$KIBANA_PORT/api/saved_objects/_export?type=dashboard&type=index-pattern&type=visualization" \
  -o kibana-export.ndjson
```

Import saved objects:

```bash
curl -sf --cacert config/certs/ca/ca.crt \
  -u elastic:$ELASTIC_PASSWORD \
  -H "kbn-xsrf: true" \
  -F "file=@kibana-export.ndjson" \
  "https://localhost:$KIBANA_PORT/api/saved_objects/_import?overwrite=true"
```
