# Elasticsearch

This directory contains the build context for the custom Elasticsearch image used in this stack: the Dockerfile, and the `elasticsearch.yml` configuration file shared across all three nodes.

## Directory Structure

```
elasticsearch/
  Dockerfile            # custom image build (plugins, etc.)
  config/
    elasticsearch.yml   # shared node configuration
```

## How the Cluster is Structured

The stack runs three Elasticsearch nodes: `es-01`, `es-02`, `es-03`. All three are master-eligible and hold data. There is no dedicated master-only node at this scale — three nodes is the minimum for a quorum-based cluster that can tolerate one node failure without losing availability.

Node-specific settings (node name, certificate paths, port bindings) are passed as environment variables in `elk-multi-node-cluster.yml`. Settings that are identical across all nodes live in `elasticsearch.yml`.

## Key Configuration Decisions

### Memory

```yaml
bootstrap.memory_lock: true
ES_JAVA_OPTS: -Xms1g -Xmx1g
```

Memory lock prevents the OS from swapping Elasticsearch heap to disk, which would cause severe latency spikes. The JVM heap is set to a fixed size (same min and max) to avoid heap resizing pauses at runtime. The heap size is configured in `.env` via `ES_HEAP_SIZE` and defaults to `1g`. As a rule, never set heap above 50% of available RAM and never above 31g (beyond that, JVM loses compressed pointer optimization).

The host must have `vm.max_map_count=262144` set or Elasticsearch will refuse to start. See the root `README.md` for the setup command.

### Security

TLS is enabled on both the HTTP layer (client-to-node) and the transport layer (node-to-node). Certificates are generated by the `elastic-setup` service on first run and stored in the `certs` Docker volume, which is mounted read-only into each node.

```yaml
xpack.security.enabled: true
xpack.security.http.ssl.enabled: true
xpack.security.transport.ssl.enabled: true
xpack.security.transport.ssl.verification_mode: certificate
```

`verification_mode: certificate` on transport means nodes verify each other's certificates against the shared CA, but do not enforce hostname matching. This is appropriate for container environments where hostnames are predictable but not necessarily embedded in the cert's SAN.

### Discovery

```yaml
cluster.initial_master_nodes: es-01,es-02,es-03
discovery.seed_hosts: es-02,es-03   # (varies per node via env var)
```

`cluster.initial_master_nodes` is only used on the very first cluster bootstrap. Once the cluster is formed and state is persisted to disk, this setting is ignored on subsequent restarts. It should not be removed from the config but it has no effect after initial formation.

### Logging

Log level is set to `warn` in production to reduce noise. Change to `info` temporarily when debugging cluster issues:

```bash
# Without restarting — applies immediately
curl -X PUT https://localhost:9200/_cluster/settings \
  -u elastic:$ELASTIC_PASSWORD \
  --cacert config/certs/ca/ca.crt \
  -H "Content-Type: application/json" \
  -d '{"transient": {"logger._root": "info"}}'
```

## Useful Commands

Check cluster health:

```bash
curl -s https://localhost:9200/_cluster/health?pretty \
  -u elastic:$ELASTIC_PASSWORD \
  --cacert config/certs/ca/ca.crt
```

List all nodes and their roles:

```bash
curl -s https://localhost:9200/_cat/nodes?v \
  -u elastic:$ELASTIC_PASSWORD \
  --cacert config/certs/ca/ca.crt
```

Check shard allocation (useful when status is yellow):

```bash
curl -s https://localhost:9200/_cat/shards?v \
  -u elastic:$ELASTIC_PASSWORD \
  --cacert config/certs/ca/ca.crt
```

## Resource Limits

Each node is configured with a memory limit in Docker (`ES_MEM_LIMIT`, default `2g`). The heap (`ES_HEAP_SIZE`) should be set to roughly half the container memory limit — for example, `1g` heap with a `2g` container limit.

If a node is killed by the OOM killer, it is almost always because the heap is set too high relative to the container memory limit, leaving insufficient room for off-heap memory used by Lucene's file system cache.
