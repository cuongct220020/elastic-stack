# Fleet Server

Fleet Server is the central control plane for Elastic Agent management. It sits between Kibana and all enrolled Elastic Agents, distributing policies and collecting agent status.

## How It Works

```
Kibana (policy management)
        |
        v
Fleet Server  <-- agents enroll here and receive policies
        |
        v
Elastic Agent(s)  -- execute the policy (collect logs, metrics, etc.)
```

When you create or update an agent policy in Kibana, Kibana pushes the change to Fleet Server. Fleet Server then distributes the updated policy to all agents enrolled under that policy. Agents check in with Fleet Server periodically to receive updates and report their health status.

Fleet Server itself also collects its own metrics and ships them to Elasticsearch, which is why it requires an enrollment token and connects to Elasticsearch directly — not just to Kibana.

## Authentication — Service Token

Fleet Server authenticates to Elasticsearch using a **service token**, not a username and password. A service token is a long-lived bearer token scoped specifically to Fleet Server operations.

The token is generated once during initial setup:

```bash
# Generate a Fleet Server service token
curl -X POST https://localhost:9200/_security/service/elastic/fleet-server/credential/token/fleet-server-token \
  -u elastic:$ELASTIC_PASSWORD \
  --cacert config/certs/ca/ca.crt
```

The token value is stored in `.env` as `FLEET_SERVER_SERVICE_TOKEN` and passed to the container as an environment variable. It does not expire automatically but can be rotated using `scripts/rotate_fleet_server_token.sh`.

If Fleet Server fails to start with an authentication error, the service token is the first thing to check.

## TLS

Fleet Server exposes port `8220` for agent enrollment and check-ins. This port is secured with TLS using the certificate generated by `elastic-setup` (`fleet-server.crt` / `fleet-server.key`).

Elastic Agents connecting to Fleet Server verify the server's certificate against the shared CA (`ca.crt`). This is why the `certs` volume must be mounted into both the Fleet Server and Elastic Agent containers.

## Key Environment Variables

| Variable | Purpose |
|---|---|
| `FLEET_SERVER_SERVICE_TOKEN` | Authenticates Fleet Server to Elasticsearch |
| `FLEET_SERVER_POLICY_ID` | The Fleet Server policy defined in Kibana |
| `FLEET_SERVER_ELASTICSEARCH_HOST` | Elasticsearch node Fleet Server writes to |
| `KIBANA_FLEET_HOST` | Kibana address used for initial Fleet setup |
| `FLEET_SERVER_CERT` / `FLEET_SERVER_CERT_KEY` | TLS certificate for the enrollment endpoint |

## Health Check

Fleet Server exposes a status endpoint at `https://localhost:8220/api/status`. The Docker health check polls this endpoint and waits for a `HEALTHY` response before marking the service ready.

Because Fleet Server depends on Kibana (for initial Fleet setup) and Kibana depends on all three Elasticsearch nodes, Fleet Server has the longest startup dependency chain in the stack. Allow up to 2–3 minutes on first boot.

```bash
# Check Fleet Server status manually
curl -sf --cacert config/certs/ca/ca.crt \
  https://localhost:8220/api/status
```

## Useful Commands

List all enrolled agents via the Elasticsearch API:

```bash
curl -s https://localhost:9200/.fleet-agents/_search?pretty \
  -u elastic:$ELASTIC_PASSWORD \
  --cacert config/certs/ca/ca.crt
```

Check Fleet Server logs:

```bash
docker compose -f fleet-compose.yml logs fleet-server --tail=50
```

Rotate the service token if it is compromised or after a security review:

```bash
bash scripts/rotate_fleet_server_token.sh
```
